# Requirements Document

## Introduction

Sprint 12 implements Luma's **PWA + Polish + QA** layer — the final sprint that transforms the app into a fully installable, offline-capable Progressive Web App with optimized performance, polished microinteractions, and validated mobile viewport compatibility. Per `BUILD_PLAN §19`, `TECHNICAL_ARCHITECTURE §21/§23`, and `PRD §13/§14`, this sprint ensures the app is ready for daily use: installable from the browser, operational offline for all locally-stored finance data (transactions, budgets, savings, reports), lazy-loaded for heavy modules (reports/charts, AI), and smooth across QA viewports (360px, 390px, 430px, 480px). All finance features must continue working fully offline. Only AI features (Gemini API calls) degrade gracefully when offline. No new npm dependencies are introduced — `vite-plugin-pwa` is already installed from Sprint 0. No push notifications, cloud sync, background sync, or login are added.

## Glossary

- **PWA_Manifest**: The web app manifest configuration provided to `vite-plugin-pwa` in `vite.config.ts` that enables browser install prompts.
- **Service_Worker**: The Workbox-based service worker generated by `vite-plugin-pwa` that handles precaching and runtime caching.
- **App_Shell**: The minimal HTML, CSS, and JavaScript required to render the app's UI skeleton (layout, BottomNav, PageWrapper) without network access.
- **Precache**: The set of static assets (HTML, JS chunks, CSS, fonts, character images) cached during service worker installation.
- **Runtime_Cache**: The caching strategy applied to assets not in the precache manifest (e.g., dynamically imported chunks).
- **Install_Prompt**: The browser's `beforeinstallprompt` event that Luma intercepts to show a custom soft install banner.
- **Install_Banner**: A dismissible, non-intrusive UI component that invites users to install the PWA.
- **Offline_Indicator**: A subtle toast or badge displayed when the app detects no network connectivity.
- **Lazy_Module**: A route or feature chunk loaded via dynamic `import()` only when needed (reports, charts, AI parser, AI insights, export).
- **Code_Split_Chunk**: A separate JavaScript bundle produced by Vite's build-time code splitting for a lazy module.
- **Touch_Target**: Any interactive element (button, link, card, icon) that must meet a minimum tap size for mobile usability.
- **Viewport_QA**: The process of verifying layout correctness at 360px, 390px, 430px, and 480px widths.
- **Microinteraction**: A subtle Framer Motion animation applied to page transitions, bottom sheets, toasts, FAB, and card interactions.
- **Page_Transition**: The animated enter/exit applied when navigating between routes.
- **vite-plugin-pwa**: The already-installed Vite plugin that generates the service worker and manifest at build time.
- **Workbox**: The underlying library used by vite-plugin-pwa for service worker caching strategies.
- **beforeinstallprompt**: The browser event fired when PWA install criteria are met, intercepted to defer the native prompt.
- **StaleWhileRevalidate**: A Workbox runtime caching strategy that serves from cache while fetching an update in the background.
- **CacheFirst**: A Workbox runtime caching strategy that serves from cache and only fetches from network on cache miss.
- **NetworkFirst**: A Workbox runtime caching strategy that tries network first, falling back to cache on failure.

## Requirements

### Requirement 1: PWA Manifest Configuration

**User Story:** As a user, I want to install Luma on my home screen like a native app, so that I can access it instantly without opening the browser.

#### Acceptance Criteria

1. THE PWA_Manifest SHALL include `name` set to "Luma", `short_name` set to "Luma", and `description` set to a concise Indonesian description of the app.
2. THE PWA_Manifest SHALL set `display` to `"standalone"` and `orientation` to `"portrait"`.
3. THE PWA_Manifest SHALL set `start_url` to `"/"` and `scope` to `"/"`.
4. THE PWA_Manifest SHALL include `theme_color` and `background_color` values matching the Cozy Dark default theme (`#1A1410`).
5. THE PWA_Manifest SHALL include an `icons` array with at least: a 192x192 PNG icon, a 512x512 PNG icon, and a 512x512 maskable PNG icon.
6. THE PWA_Manifest SHALL be generated via `vite-plugin-pwa` configuration in `vite.config.ts`, not as a separate static file.
7. WHEN the built app is audited by Chrome DevTools Lighthouse, THE PWA_Manifest SHALL satisfy all installability criteria (valid manifest, registered service worker, HTTPS or localhost).

---

### Requirement 2: Service Worker Registration and Precaching

**User Story:** As a user, I want the app to cache itself on first visit, so that subsequent opens are instant and work without internet.

#### Acceptance Criteria

1. THE vite-plugin-pwa configuration SHALL use `registerType: "autoUpdate"` so the service worker updates silently without user intervention.
2. THE Service_Worker SHALL precache the App_Shell: the root HTML document, all CSS bundles, the main JavaScript entry chunk, font files, and all character asset images in `public/characters/`.
3. THE Service_Worker SHALL use Workbox's `cleanupOutdatedCaches: true` to remove stale precache entries on update.
4. WHEN the Service_Worker activates for the first time, THE Service_Worker SHALL claim all open clients immediately via `clientsClaim: true`.
5. THE vite-plugin-pwa configuration SHALL include a `workbox.globPatterns` that matches `**/*.{js,css,html,ico,png,webp,woff2}`.
6. THE Service_Worker SHALL include a navigation fallback to `index.html` for all navigation requests (SPA routing support).
7. IF the Service_Worker fails to register (unsupported browser), THEN THE app SHALL continue to function normally without caching and SHALL NOT show an error to the user.

---

### Requirement 3: Runtime Caching Strategy

**User Story:** As a user, I want dynamically loaded features (reports, AI) to be cached after first use, so that they load quickly on repeat visits.

#### Acceptance Criteria

1. THE Service_Worker SHALL apply a CacheFirst strategy with a maximum age of 30 days to all image assets matching `**/*.{png,webp,jpg,svg}`.
2. THE Service_Worker SHALL apply a StaleWhileRevalidate strategy to all JavaScript and CSS chunks not in the precache manifest (lazy-loaded code-split chunks).
3. THE Service_Worker SHALL apply a NetworkFirst strategy with a 3-second timeout to any Gemini API requests, falling back to a cached response when offline.
4. THE Service_Worker SHALL NOT cache IndexedDB data (IndexedDB operates independently of the service worker cache).
5. WHEN a runtime-cached asset exceeds its maximum cache entries (50 entries per cache), THE Service_Worker SHALL evict the least-recently-used entry.

---

### Requirement 4: Offline Finance Functionality

**User Story:** As a user, I want to record transactions, check budgets, and view savings goals even without internet, so that the app is reliable on flights or in areas with no signal.

#### Acceptance Criteria

1. WHILE the device has no network connectivity, THE app SHALL allow creating, editing, and deleting transactions via IndexedDB without error.
2. WHILE the device has no network connectivity, THE app SHALL allow viewing and editing budgets (monthly and category) via IndexedDB without error.
3. WHILE the device has no network connectivity, THE app SHALL allow creating saving goals and adding contributions via IndexedDB without error.
4. WHILE the device has no network connectivity, THE app SHALL allow viewing reports and charts using locally stored transaction data without error.
5. WHILE the device has no network connectivity, THE app SHALL allow exporting spreadsheets (XLSX/CSV) and visual reports (PDF) using locally stored data without error.
6. WHILE the device has no network connectivity, THE app SHALL allow full customization (theme switching, character switching, background upload) without error.
7. THE app SHALL NOT show network-related errors for any finance operation that uses only IndexedDB.

---

### Requirement 5: Offline AI Degradation

**User Story:** As a user, I want to know clearly when AI features are unavailable offline, so that I can still use manual input without confusion.

#### Acceptance Criteria

1. WHILE the device has no network connectivity, THE AI Quick Input tab SHALL display a soft message "AI butuh internet — pakai manual dulu ya ✨" and SHALL disable the AI parse button.
2. WHILE the device has no network connectivity, THE AI Behavioral Insights section on the Reports page SHALL display "Insight AI butuh koneksi internet 🌐" instead of attempting a Gemini API call.
3. WHILE the device has no network connectivity, THE manual transaction form SHALL remain fully functional and accessible as the primary input method.
4. WHEN network connectivity is restored, THE AI features SHALL become available again without requiring a page refresh or app restart.
5. THE app SHALL detect connectivity changes using the `navigator.onLine` API combined with the `online`/`offline` window events.

---

### Requirement 6: Offline Status Indicator

**User Story:** As a user, I want a subtle indicator when I'm offline, so that I understand why AI features are disabled without feeling alarmed.

#### Acceptance Criteria

1. WHEN the device transitions from online to offline, THE Offline_Indicator SHALL display a non-blocking toast with the message "Kamu lagi offline — data tetap aman kok 🔒" for 3 seconds.
2. WHEN the device transitions from offline to online, THE Offline_Indicator SHALL display a non-blocking toast with the message "Online lagi ✨" for 2 seconds.
3. THE Offline_Indicator toasts SHALL use the existing Toast component and SHALL NOT block user interaction.
4. THE Offline_Indicator SHALL NOT display on initial app load if the device is already offline (only transition events trigger the toast).

---

### Requirement 7: Install Prompt UX

**User Story:** As a user, I want a friendly, non-intrusive prompt to install the app, so that I can add it to my home screen without being nagged.

#### Acceptance Criteria

1. WHEN the browser fires `beforeinstallprompt`, THE app SHALL intercept and store the event (preventing the default browser mini-infobar).
2. THE Install_Banner SHALL appear at the top of the HomePage as a dismissible card with the message "Pasang Luma di home screen biar makin cepat dibuka ✨" and a primary button "Pasang".
3. WHEN the user taps "Pasang", THE Install_Banner SHALL invoke the stored `beforeinstallprompt` event's `prompt()` method.
4. WHEN the user dismisses the Install_Banner (taps the close icon), THE app SHALL store a flag in localStorage (`installBannerDismissed: true`) and SHALL NOT show the banner again for that device.
5. THE Install_Banner SHALL NOT appear if the app is already running in standalone mode (`window.matchMedia('(display-mode: standalone)').matches === true`).
6. THE Install_Banner SHALL NOT appear if `installBannerDismissed` is `true` in localStorage.
7. THE Install_Banner SHALL render with a subtle enter animation (fade + slide down, 200ms duration) using Framer Motion.

---

### Requirement 8: Lazy Loading Heavy Modules

**User Story:** As a user, I want the app to open fast by loading only what I need first, so that the initial experience feels snappy.

#### Acceptance Criteria

1. THE ReportsPage component SHALL be loaded via `React.lazy()` with a dynamic `import()` so it is in a separate Code_Split_Chunk.
2. THE AI parser module (Gemini client and parsing logic) SHALL be loaded via dynamic `import()` only when the user activates the AI Quick Input tab.
3. THE AI behavioral insights module SHALL be loaded via dynamic `import()` only when the user navigates to the Reports page and requests AI insights.
4. THE export modules (jsPDF, html2canvas, xlsx) SHALL be loaded via dynamic `import()` only when the user initiates a download/export action.
5. THE Recharts library SHALL be loaded via dynamic `import()` as part of the ReportsPage lazy chunk, not in the main bundle.
6. WHILE a lazy module is loading, THE app SHALL display a lightweight loading skeleton or spinner (not a blank screen).
7. IF a lazy module fails to load (network error, chunk not cached), THEN THE app SHALL display a soft error message "Gagal memuat modul, coba refresh ya" with a retry button.

---

### Requirement 9: Image Optimization

**User Story:** As a user, I want images in the app to load quickly and not drain my storage, so that the app stays lightweight.

#### Acceptance Criteria

1. ALL character assets in `public/characters/` SHALL be in WebP format and each file SHALL be at most 30KB in size.
2. ALL icon assets in `public/icons/` SHALL be optimized PNG files with no unnecessary metadata.
3. THE PWA icons (192x192, 512x512) SHALL be compressed PNG files totaling no more than 100KB combined.
4. THE app SHALL NOT load character assets that are not currently active (only the selected character's mood states are loaded).
5. WHEN background images are uploaded by the user, THE existing imageProcessor from Sprint 9 SHALL handle compression (resize ≤ 1080px, WebP, quality 0.85).

---

### Requirement 10: Mobile Viewport Compatibility

**User Story:** As a user on various phone sizes, I want the app to look correct on my screen without horizontal scrolling or broken layouts.

#### Acceptance Criteria

1. THE app layout SHALL NOT produce horizontal overflow (no horizontal scrollbar) at viewport widths of 360px, 390px, 430px, and 480px.
2. THE BottomNav SHALL remain fixed at the bottom and fully visible at all QA viewports without overlapping content.
3. THE HomePage sections (budget card, character area, quick stats, recent transactions, FAB) SHALL stack vertically without overlap at all QA viewports.
4. THE AddTransactionSheet and all bottom sheets SHALL not exceed the viewport width at 360px.
5. ALL text content SHALL remain readable (no truncation that hides critical financial data like amounts or category names) at 360px viewport width.
6. THE TransactionsPage filter row SHALL wrap or scroll horizontally (within its container) without breaking the page layout at 360px.
7. THE ReportsPage charts SHALL resize responsively to fit within the content area at all QA viewports without horizontal overflow.

---

### Requirement 11: Touch Target Accessibility

**User Story:** As a user with average-sized fingers, I want all buttons and interactive elements to be easy to tap, so that I don't accidentally trigger the wrong action.

#### Acceptance Criteria

1. ALL interactive elements (buttons, icon buttons, links, cards, nav items, form controls) SHALL have a minimum touch target size of 44×44 CSS pixels.
2. THE BottomNav items SHALL each have a minimum tap area of 44px height.
3. THE FAB (Floating Action Button) SHALL have a minimum size of 56×56 CSS pixels.
4. THE bottom sheet action buttons SHALL have a minimum height of 44px.
5. WHEN interactive elements are spaced adjacently, THE gap between touch targets SHALL be at least 8px to prevent mis-taps.

---

### Requirement 12: Semantic HTML and ARIA Labels

**User Story:** As a user who might use assistive technology, I want the app to use proper semantic markup, so that screen readers can navigate it.

#### Acceptance Criteria

1. THE BottomNav SHALL use a `<nav>` element with `aria-label="Navigasi utama"`.
2. ALL icon-only buttons (settings gear, close buttons, FAB) SHALL have an `aria-label` describing their action.
3. THE bottom sheets SHALL use `role="dialog"` with `aria-modal="true"` and an `aria-labelledby` pointing to the sheet title.
4. THE toast notifications SHALL use `role="status"` and `aria-live="polite"`.
5. ALL form inputs SHALL have associated `<label>` elements or `aria-label` attributes.
6. THE page headings SHALL follow a logical heading hierarchy (h1 for page title, h2 for sections).
7. THE app SHALL NOT use `div` or `span` with click handlers without an appropriate `role` and keyboard accessibility (`tabIndex`, `onKeyDown`).

---

### Requirement 13: Focus Management

**User Story:** As a user navigating with keyboard or assistive technology, I want focus to move logically when modals open and close, so that I don't lose my place.

#### Acceptance Criteria

1. WHEN a bottom sheet opens, THE focus SHALL move to the first focusable element inside the sheet.
2. WHEN a bottom sheet is open, THE focus SHALL be trapped within the sheet (Tab cycling stays inside the dialog).
3. WHEN a bottom sheet closes, THE focus SHALL return to the element that triggered the sheet.
4. WHEN a toast appears, THE focus SHALL NOT move to the toast (it is announced via `aria-live` only).
5. THE BottomNav items SHALL be focusable and activatable via keyboard (Enter/Space).

---

### Requirement 14: Color Contrast

**User Story:** As a user, I want text to be readable against all theme backgrounds, so that I can use the app comfortably in any theme.

#### Acceptance Criteria

1. ALL text rendered on card surfaces SHALL maintain a contrast ratio of at least 4.5:1 against the card background color in all 5 preset themes.
2. ALL text rendered on the main page background SHALL maintain a contrast ratio of at least 4.5:1 against the background color in all 5 preset themes.
3. WHEN a custom background image is active, THE overlay (minimum opacity 0.30 per Sprint 9 constraints) SHALL ensure body text remains readable.
4. THE accent-colored interactive elements (buttons, links) SHALL maintain a contrast ratio of at least 3:1 against their surrounding background.

---

### Requirement 15: Page Transition Animations

**User Story:** As a user, I want smooth transitions between pages, so that navigation feels polished and spatial.

#### Acceptance Criteria

1. WHEN navigating between routes, THE outgoing page SHALL animate out with a fade + subtle scale-down (opacity 1→0, scale 1→0.98) over 150ms using Framer Motion.
2. WHEN navigating between routes, THE incoming page SHALL animate in with a fade + subtle scale-up (opacity 0→1, scale 0.98→1) over 200ms using Framer Motion.
3. THE page transitions SHALL use `AnimatePresence` from Framer Motion with `mode="wait"` to prevent layout jumps.
4. THE page transitions SHALL NOT block user interaction for longer than 250ms total.
5. THE page transitions SHALL use `will-change: opacity, transform` to hint GPU acceleration and avoid jank.

---

### Requirement 16: Bottom Sheet Animations

**User Story:** As a user, I want bottom sheets to slide up and down smoothly, so that the interaction feels natural like a native app.

#### Acceptance Criteria

1. WHEN a bottom sheet opens, THE sheet SHALL animate from `translateY(100%)` to `translateY(0)` over 250ms with an ease-out curve using Framer Motion.
2. WHEN a bottom sheet closes, THE sheet SHALL animate from `translateY(0)` to `translateY(100%)` over 200ms with an ease-in curve using Framer Motion.
3. WHEN a bottom sheet opens, THE backdrop overlay SHALL fade in from opacity 0 to the configured overlay opacity over 200ms.
4. WHEN a bottom sheet closes, THE backdrop overlay SHALL fade out over 150ms.
5. THE bottom sheet animation SHALL NOT cause layout reflow on the underlying page content.

---

### Requirement 17: Toast Animations

**User Story:** As a user, I want toast messages to appear and disappear gently, so that they inform without startling me.

#### Acceptance Criteria

1. WHEN a toast appears, THE toast SHALL animate in with a slide-up + fade (translateY(20px)→0, opacity 0→1) over 200ms using Framer Motion.
2. WHEN a toast disappears, THE toast SHALL animate out with a fade (opacity 1→0) over 150ms.
3. THE toast SHALL auto-dismiss after its configured duration (default 3 seconds) unless otherwise specified.
4. WHEN multiple toasts are queued, THE toast system SHALL show them sequentially (one at a time), not stacked.

---

### Requirement 18: FAB and Card Interaction Animations

**User Story:** As a user, I want subtle feedback when I tap the FAB or interact with cards, so that the app feels alive and responsive.

#### Acceptance Criteria

1. WHEN the user taps the FAB, THE FAB SHALL animate with a brief scale pulse (1→0.9→1) over 150ms using Framer Motion.
2. WHEN a transaction card or budget card is tapped, THE card SHALL animate with a subtle press effect (scale 1→0.97→1) over 120ms.
3. THE FAB SHALL have a subtle idle animation: a gentle floating motion (translateY oscillation of 2px) with a 3-second loop, using Framer Motion `animate` with `repeat: Infinity`.
4. ALL microinteraction animations SHALL respect `prefers-reduced-motion: reduce` — WHEN the user has reduced motion enabled, THE animations SHALL be disabled (instant state changes instead).

---

### Requirement 19: Performance Budget

**User Story:** As a user, I want the app to open fast and stay smooth, so that it feels like a native app.

#### Acceptance Criteria

1. THE initial JavaScript bundle (main chunk excluding lazy modules) SHALL be at most 200KB gzipped.
2. THE total precache size (all files in the service worker precache manifest) SHALL be at most 2MB uncompressed.
3. THE app SHALL achieve a Lighthouse Performance score of at least 90 on mobile simulation (Moto G Power, slow 4G) for the Home page.
4. ALL Framer Motion animations SHALL run at 60fps without dropped frames on a mid-range device (no heavy blur, no complex path animations, no large layout animations).
5. THE derived calculations (monthly totals, budget usage, category totals) SHALL be memoized so they recompute only when their input transactions change.

---

### Requirement 20: No Scope Creep

**User Story:** As a developer, I want clear boundaries on what this sprint does NOT include, so that implementation stays focused.

#### Acceptance Criteria

1. THE Sprint 12 implementation SHALL NOT add push notification support.
2. THE Sprint 12 implementation SHALL NOT add cloud sync or any remote data persistence.
3. THE Sprint 12 implementation SHALL NOT add background sync capabilities.
4. THE Sprint 12 implementation SHALL NOT add user authentication or login.
5. THE Sprint 12 implementation SHALL NOT introduce any new npm dependencies beyond what is already in `package.json`.
6. THE Sprint 12 implementation SHALL NOT modify the IndexedDB schema or add new object stores.
7. THE Sprint 12 implementation SHALL NOT change the existing BottomNav structure (Home, Transaksi, Target, Laporan).
